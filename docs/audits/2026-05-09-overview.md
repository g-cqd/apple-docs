# Audit 1 — Overview Review

**Date:** 2026-05-09
**Scope:** broad pass on security, performance, code quality and architecture.

The apple-docs codebase is an exemplary example of high-performance JavaScript engineering, specifically optimized for the Bun runtime. It demonstrates a deep understanding of database internals, systems programming, and network resilience.

## Security Review

- **SQL Injection (Excellent):** The project consistently uses parameterized queries via `bun:sqlite`'s `$param` syntax. There is no evidence of string-concatenated SQL, providing robust protection against injection.
- **Path Traversal (Low Risk / Strong Defense):**
  - Document paths are resolved via `src/lib/safe-path.js`, which handles deterministic shortening for filesystem limits.
  - While the lower-level `keyPath` utility doesn't explicitly block `..` segments, the lookup logic gates all file access behind a database check. Since the database keys are populated from trusted Apple API discovery, the system is effectively protected.
  - **Recommendation:** Implement "defense in depth" by adding a check in `src/lib/safe-path.js` to ensure keys do not contain `..` or attempt to escape the `dataDir`.
- **Input Validation (Strong):** The primary entry point (MCP Server) uses Zod for rigorous schema validation. All tool arguments are coerced and validated before reaching internal logic.
- **Networking:** The system is a "polite citizen," implementing hostname-bucketed token-bucket rate limiting and strictly honoring `Retry-After` headers.

## Performance Review

- **Storage Layer (Elite):**
  - **Memory-Mapped I/O:** Configures `PRAGMA mmap_size = 10GB` to allow zero-syscall FTS5 index scans, essentially treating the disk as virtual memory.
  - **Concurrency:** Uses WAL (Write-Ahead Logging) mode and `synchronous = NORMAL` for high-throughput concurrent reads and writes.
  - **Search:** Implements both FTS5 (for full-text) and Trigram indexes (for fuzzy title matching), ensuring sub-millisecond lookups even on large corpora.
- **Static Site Generation (Exceptional):**
  - **Parallelism:** Uses bin-packed fan-out across multiple Bun subprocesses, ensuring all CPU cores are utilized efficiently during rendering.
  - **Incremental Builds:** Employs a clever "shape-based" fingerprinting (hashing section lengths/kinds rather than full content) to make the "should I re-render?" check extremely fast.
  - **Compression:** Pre-compresses assets with Brotli (Quality 11) during build time, offloading the CPU cost from the web server to the CI/CD pipeline.
- **Batched I/O:** The `batchFetchSections` utility prevents N+1 query problems by chunking thousands of document lookups into a handful of batched SQL calls.

## Code Quality & Architecture

- **Engineering Standards:** The code is modular, clean, and follows modern ES module patterns. It uses `tsc --noEmit` for type safety while maintaining the flexibility of JavaScript.
- **Robustness:**
  - **Transactions:** A clean `tx()` helper ensures atomic DB operations and automatic rollbacks.
  - **Atomic Swaps:** The web build uses an atomic directory rename to ensure users never see a partially rendered site.
  - **Migrations:** A versioned migration system (currently at schema v13) provides a clear path for database evolution.
- **Tooling:** First-class support for benchmarking (`bun run bench`), mutation testing (Stryker), and duplication detection (jscpd).

**Conclusion:** This is a production-grade codebase with a heavy emphasis on stability and extreme performance. The few identified risks are minor and mitigated by high-level architectural patterns.
